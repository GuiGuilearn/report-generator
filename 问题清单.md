# 问题清单

## 1. 课程代码与运行结果混在同一张图片中

- 表现：第四部分“课程代码及执行结果”里，PNG 同时包含源代码和运行输出，不符合“代码必须是文字、结果必须是图片”的报告要求。
- 原因：`render_code_result()` 原设计负责渲染“任务名称 + 代码 + 输出 + 图形”，模板填充逻辑只插入图片，没有单独插入成功任务的源代码文字。
- 处理：将渲染器改为只渲染运行输出和可选图形；在 Word 填充阶段为每个任务插入“源代码：”文字段落，再插入“运行结果：”图片。
- 状态：已解决。

## 10. DeepSeek 阶段二仍返回非标准 JSON，导致高铁售票任务无源代码

- 表现：高铁售票任务在 PowerShell 中仍报“LLM 返回非标准 JSON，无法生成代码”，Word 第四部分显示“未生成源代码”，需要人工补全。
- 原因：DeepSeek 的 OpenAI-compatible 接口即使传入 `response_format={"type":"json_object"}`，也不能保证复杂代码任务一定返回可被 `json.loads()` 解析的完整 JSON；同时原逻辑把“JSON 解析失败”直接等同于“代码生成失败”，没有继续从普通文本中提取 Python 源码，也没有本地保底实现。
- 处理：阶段二改为四级兜底：先用 JSON 模式生成并放大 `max_tokens`，失败后重试；仍失败时改用“只返回 Python 源代码”的普通文本模式，并自动抽取代码块、用 `ast.parse()` 校验；若命中“高铁/售票/座位/20/5”等课堂任务关键词，则使用本地稳定模板生成完整可运行代码和测试输入。
- 状态：已解决。

## 2. 输出结果图片中中文乱码或显示异常

- 表现：PNG 中的中文输出存在乱码、方块或无法正确显示。
- 原因：字体候选列表主要是 Linux 字体族名，Windows 环境下 Pillow 无法通过这些名称加载中文字体，退回默认字体后缺少中文字形。
- 处理：渲染器优先按 Windows 字体文件路径加载 `NotoSansSC-VF.ttf`、`msyh.ttc`、`simsun.ttc`、`simhei.ttf`，再尝试字体族名称和 Linux fallback。
- 状态：已解决。

## 3. 开发文档与实际实现不一致

- 表现：`.uploads` 中开发文档仍描述“代码 + 输出 + 图形渲染为 PNG”，并记录 `/tmp/_output_figure.png` 等旧路径。
- 原因：实现经过多轮修正后，文档没有同步更新。
- 处理：更新图片渲染器、模板引擎、代码与结果分离、字体 fallback、图形输出路径和变更记录相关说明。
- 状态：已解决。

## 4. 未设置 API Key 时脚本直接抛出 OpenAI SDK 异常

- 表现：启动后在 `OpenAI(...)` 初始化处报 `Missing credentials` traceback。
- 原因：脚本只读取 `OPENAI_API_KEY`，没有支持 DeepSeek 专用环境变量，也没有在初始化客户端前做参数校验。
- 处理：新增 `--provider`、`--api-key`、`--base-url`、`--model` 参数；支持 `config.yaml` 中的 `llm.providers`；默认读取 `DEEPSEEK_API_KEY`，同时保留 `OPENAI_API_KEY` fallback；缺 key 时输出友好错误并退出。
- 状态：已解决。

## 5. Windows 用户变量设置后当前 PowerShell 不生效

- 表现：已在 Windows“用户变量”中设置 `DEEPSEEK_API_KEY`，但当前 PowerShell 仍读取不到，或继续使用旧值。
- 原因：Windows 环境变量只会在新进程启动时读取；已经打开的 PowerShell 不会自动刷新用户变量。
- 处理：提示用户关闭并重新打开 PowerShell，再用 `echo $env:DEEPSEEK_API_KEY` 检查是否生效。
- 状态：已记录。

## 6. API Key 截图泄露风险

- 表现：截图中完整显示了 DeepSeek API Key。
- 原因：环境变量窗口会明文显示变量值，截图或分享时容易泄露。
- 处理：提醒用户重置/重新生成 key，并避免在截图、文档、聊天记录中暴露完整 key。
- 状态：已记录。

## 7. Word 中源代码格式与手动补全需求不匹配

- 表现：曾将源代码改为等宽字体、语法高亮和浅灰背景的代码块样式，但用户手动补全失败任务时不方便保持格式。
- 原因：报告交付更需要与正文一致、易编辑的普通 Word 文本，而不是代码编辑器视觉效果。
- 处理：撤回语法高亮代码块，源代码改回与正文一致的普通文本格式；仍保留缩进和换行，便于复制到 Jupyter Notebook 运行，也便于在 Word 中手动补充。
- 状态：已解决。

## 8. 运行结果图片字号偏小

- 表现：结果图片插入 Word 后字太小，看不清。
- 原因：渲染器画布较宽但字体只有 14px，插入 Word 时再缩放到页面宽度，导致显示偏小。
- 处理：提高结果图片标题/正文内部字号，增大 padding 和行距，并将 Word 插图宽度提高到 6.3 英寸。
- 状态：已解决。

## 9. 阶段二代码生成 JSON 解析失败时报告中出现空任务

- 表现：PowerShell 中提示“LLM 返回非标准 JSON”，Word 中只显示“任务 2：任务2”“源代码为空”“LLM 生成代码为空”，无法判断失败的是哪道题。
- 原因：阶段二失败后只创建了一个空 `CodeTask`，任务名使用机械编号，没有保留原始任务描述、失败原因和模型响应摘要；JSON 解析也只接受纯 JSON，不能处理模型在 JSON 外包裹说明文字的情况。
- 处理：增强 JSON 解析，纯 JSON 解析失败时会尝试截取第一个 `{` 到最后一个 `}` 之间的内容再次解析；失败任务使用原始任务描述生成可读标题，并在报告中写入“任务描述”“未生成源代码”“失败原因”和 LLM 原始响应摘要。
- 状态：已解决。
