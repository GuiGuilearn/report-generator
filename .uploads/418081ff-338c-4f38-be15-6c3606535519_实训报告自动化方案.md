# 实训报告自动生成系统 — 开发文档

## 1. 项目概述

### 1.1 项目背景

用户每日需要将老师提供的上课笔记（TXT 格式）整理为一套标准格式的实训报告（DOCX 格式），报告包含四个固定章节：课程目标、课程内容（小标题列表）、课程内容详情、代码及运行结果。整个过程完全手工操作，耗时重复。

### 1.2 项目目标

构建一个端到端的自动化流水线，输入为 TXT 笔记文件，输出为填充完整的 DOCX 报告。系统内部调用大语言模型完成语义理解与内容生成，通过 Jupyter 运行时执行代码并捕获输出（含图形输出），最终将结构化数据注入 Word 模板。

### 1.3 技术栈

| 层级 | 技术选型 | 版本要求 |
|------|----------|----------|
| 运行时 | Python | >= 3.10 |
| 大模型接口 | OpenAI SDK | >= 1.0 |
| 文档处理 | python-docx | >= 0.8 |
| 代码执行 | nbformat + nbconvert | >= 7.0 |
| 图片渲染 | Pillow | >= 10.0 |
| 配置管理 | PyYAML | >= 6.0 |
| 任务调度 | TRAE Schedule | 内建 |

### 1.4 配置管理

个人信息通过 `config.yaml` 外部化，无需修改代码：

```yaml
# config.yaml
student:
  class_name: "23060101"
  group: "第二组"
  student_id: "2023032"
  name: "GuiGui"
  school: "计算机科学与工程学院"

output:
  naming: "{class_name}_{group}_{student_id}_{name}.docx"

llm:
  provider: "deepseek"
  model: "deepseek-chat"
  base_url: "https://api.deepseek.com"
  api_key_env: "DEEPSEEK_API_KEY"
  temperature: 0.3
  max_retries: 3
  providers:
    deepseek:
      model: "deepseek-chat"
      base_url: "https://api.deepseek.com"
      api_key_env: "DEEPSEEK_API_KEY"
    openai:
      model: "gpt-4.1-mini"
      base_url: "https://api.openai.com/v1"
      api_key_env: "OPENAI_API_KEY"

execution:
  timeout_per_cell: 30       # 秒，单个 Cell 执行超时
  auto_fix_max_attempts: 3   # 代码报错后自动修复的最大次数
```

## 3. 数据流设计

### 3.1 顶层数据流

```
TXT 文件
  │
  ├─(1a)─► [LLM 阶段一] ──► TextData {course_title, objectives, topics, details}
  │
  ├─(1b)─► [LLM 阶段二] ──► code_tasks[] （每个任务独立调用，互不影响）
  │                              │
  │                     ┌────────┴────────┐
  │                     │                 │
  │                text_fields      code_tasks[]
  │                     │                 │
  │                     │           ┌─────┴──────┐
  │                     │      (2)  │ Code Sandbox│
  │                     │           │             │
  │                     │           │ AST 扫描    │
  │                     │           │ 子进程隔离  │
  │                     │           │ 自动修复循环│
  │                     │           └─────┬──────┘
  │                     │                 │
  │                     │           execution_results[]
  │                     │           (含 success/fail 状态)
  │                     │                 │
  │                     │           (3)   │
  │                     │           ┌─────┴──────┐
  │                     │           │ Renderer    │
  │                     │           └─────┬──────┘
  │                     │                 │
  │                     │           image_paths[]
  │                     │                 │
  │                     └────────┬────────┘
  │                              │
  └─(4)─► [Template Engine] ◄────┘
                   │
                   ▼
              DOCX 文件
            （部分失败也生成）
```

### 3.2 核心数据结构

```typescript
// 阶段一输出：文本内容
interface TextData {
    course_title: string;
    objectives: string;
    topics: string[];
    details: string;
    task_descriptions: string[];   // 代码任务描述列表，供阶段二使用
}

// 阶段二输出：单个代码任务
interface CodeTask {
    name: string;
    description: string;
    test_inputs: string[];         // 模拟用户输入；纯输出任务为空 []
    code: string;                  // 完整可运行代码，已含 input 替换器
}

// 执行结果
interface ExecutionResult {
    task: CodeTask;
    status: "success" | "failed";
    output_text: string;           // stdout 文本
    image_paths: string[];         // 渲染后的图片路径（可能含图形输出）
    error_message: string | null;  // 失败时的错误信息
}

// 个人/封面信息
interface StudentInfo {
    class_name: string;
    group: string;
    student_id: string;
    name: string;
    school: string;
}
```

### 3.3 各阶段数据转换

| 阶段 | 输入 | 输出 | 说明 |
|------|------|------|------|
| LLM 阶段一 | `txt_content: string` | `TextData` | 提取结构化文本，token 约 500-1000 |
| LLM 阶段二 | `task_descriptions: string[]` | `CodeTask[]` | 每个任务独立调用，互不影响 |
| 代码沙箱 | `CodeTask[]` | `ExecutionResult[]` | AST 扫描 + 子进程隔离 + 自动修复 |
| 图片渲染 | `ExecutionResult[]` | `image_paths[]` | 白底 PNG，仅含运行输出和图形；源代码保留为 Word 文本 |
| 模板填充 | `TextData + ExecutionResult[] + StudentInfo` | `.docx` 文件 | 书签定位，部分失败也生成 |

## 4. 模块详细设计

### 4.1 LLM 服务

#### 4.1.1 两阶段调用设计

**设计动机**：单次调用返回全部内容（文本 + 代码）可能导致 token 超限截断，且一个代码任务生成失败会导致全部重来。拆分为两阶段后，文本和代码解耦，代码任务独立生成、独立重试。

**阶段一：文本提取**

```python
def ai_generate_text(txt_content: str) -> TextData:
    """
    调用 LLM 提取结构化文本内容。
    """
    response = client.chat.completions.create(
        model=config.llm.model,
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT_PHASE1},
            {"role": "user", "content": txt_content}
        ],
        response_format={"type": "json_object"},
        temperature=0.3
    )
    return parse_and_validate(response)
```

**阶段二：代码任务生成**（每个任务独立调用）

```python
def ai_generate_code(task_description: str, task_index: int) -> CodeTask:
    """
    为单个代码任务调用 LLM 生成代码和 test_inputs。
    失败时独立重试，不影响其他任务。
    """
    response = client.chat.completions.create(
        model=config.llm.model,
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT_PHASE2},
            {"role": "user", "content": task_description}
        ],
        response_format={"type": "json_object"},
        temperature=0.3,
        max_tokens=4000
    )
    return parse_and_validate(response)
```

#### 4.1.2 模型配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `provider` | `"deepseek"` | 默认服务商，指向 `llm.providers` 下的配置 |
| `model` | `"deepseek-chat"` | 模型名称，通过 config.yaml 或 `--model` 覆盖 |
| `base_url` | `"https://api.deepseek.com"` | OpenAI-compatible API 地址，通过 config.yaml 或 `--base-url` 覆盖 |
| `api_key_env` | `"DEEPSEEK_API_KEY"` | API Key 环境变量名，推荐使用环境变量而不是把 key 写进配置文件 |
| `temperature` | `0.3` | 低温度保证输出稳定 |
| `response_format` | `{"type": "json_object"}` | 强制 JSON 输出 |
| `timeout` | `60` | 秒，单次 API 调用超时 |
| `max_tokens` (阶段二) | `4000` | 每个代码任务独立限制；复杂任务避免代码字段被截断 |

#### 4.1.3 环境变量

| 变量 | 必填 | 说明 |
|------|------|------|
| `DEEPSEEK_API_KEY` | DeepSeek 推荐 | DeepSeek API Key |
| `OPENAI_API_KEY` | OpenAI 或通用 fallback | OpenAI API Key；未配置 provider 专属变量时也会作为 fallback |
| `OPENAI_BASE_URL` | 否 | 临时覆盖 API 地址 |

#### 4.1.4 输出校验

1. 去除可能的 markdown 代码块包裹
2. 尝试 `json.loads()` 解析
3. 若失败，截取第一个 `{` 到最后一个 `}` 之间的内容再次解析
4. 若阶段二 JSON 仍失败，先判断是否命中已识别的课堂任务（如高铁售票模拟），命中则直接启用本地保底模板，避免 Word 中出现空源代码
5. 未命中保底模板时，改用“只返回 Python 源代码”的普通文本模式，并从代码块或正文中抽取 Python
6. 使用 `ast.parse()` 校验抽取出的代码，校验通过则生成 `CodeTask`
7. 若所有兜底均失败，报告中保留任务描述、失败原因、JSON 响应摘要和纯代码响应摘要

#### 4.1.5 TXT 预处理

在发送给 LLM 之前，对 TXT 内容做以下处理：

1. `content.rstrip()` 清理尾部空白行（实测样本末尾有 38 行空白）
2. 标注"忽略操作提示"：注释快捷键、编辑器操作等与课程知识无关的内容

#### 4.1.6 System Prompt 约束

- 输出格式：严格 JSON，字段名不可变
- 课程目标：倒推式生成，动词开头，覆盖全部知识模块
- 课程内容：仅提取知识点名称，不展开说明
- 课程详情：用自己的话重新组织，禁止照抄原文
- 代码任务：阶段二独立生成，每个任务含 test_inputs 和完整代码
- 忽略操作提示：注释快捷键、编辑器操作等与课程知识无关的内容
- 代码中禁止重新导入 `input` 或使用 `sys.stdin.readline`

完整 Prompt 模板见附录 A。

### 4.2 代码沙箱

#### 4.2.1 安全机制

LLM 生成的代码在无人审查的情况下自动执行，存在安全风险。沙箱提供三层防护：

**第一层：AST 静态扫描**

执行前用 `ast` 模块解析代码，禁止导入以下危险模块：

```
os, subprocess, sys, shutil, socket, pickle, ctypes, importlib,
multiprocessing, signal, pty, pdb, code, codeop, compileall
```

若检测到危险导入，标记该任务为 `failed`，记录到日志，不阻断其他任务。

```python
FORBIDDEN_MODULES = {
    "os", "subprocess", "sys", "shutil", "socket", "pickle",
    "ctypes", "importlib", "multiprocessing", "signal", "pty",
    "pdb", "code", "codeop", "compileall"
}

def scan_imports(code: str) -> list[str]:
    """返回代码中导入的危险模块列表"""
    try:
        tree = ast.parse(code)
        imports = set()
        for node in ast.walk(tree):
            if isinstance(node, ast.Import):
                for alias in node.names:
                    imports.add(alias.name.split('.')[0])
            elif isinstance(node, ast.ImportFrom):
                if node.module:
                    imports.add(node.module.split('.')[0])
        return [m for m in imports if m in FORBIDDEN_MODULES]
    except SyntaxError:
        return []  # 语法错误由执行阶段处理
```

**第二层：子进程隔离**

不使用 `nbconvert` 在当前进程内执行，而是用 `subprocess` 启动独立 Python 进程执行代码。子进程崩溃不影响主流程。

**第三层：超时控制**

每个 Cell 超时从 600s 降至 30s。教学级代码任务（快递费计算、字典操作等）执行时间通常不超过 1 秒。

```python
def execute_in_sandbox(code: str, timeout: int = 30) -> tuple[bool, str, str]:
    """
    在子进程中执行代码，返回 (success, stdout, stderr)。
    """
    result = subprocess.run(
        ["python3", "-c", code],
        capture_output=True, text=True,
        timeout=timeout,
        cwd="temp/"
    )
    return (
        result.returncode == 0,
        result.stdout,
        result.stderr
    )
```

#### 4.2.2 自动修复循环

LLM 生成的代码可能"语法正确但运行报错"（如快递费计算中缺少 `float()` 转换）。沙箱集成自动修复循环：

```
for attempt in range(3):
    success, stdout, stderr = execute_in_sandbox(code)
    if success:
        break
    code = llm_fix(code, stderr)  # 将错误信息反馈给 LLM，要求修复
```

修复专用 Prompt（附录 A.2）要求 LLM 仅修复错误，不改变原有逻辑。

#### 4.2.3 部分成功机制

每个代码任务独立执行，失败的任务标记 `status: "failed"` 并记录错误信息，不阻断其他任务。

- 全部任务成功 → 退出码 0
- 部分任务失败 → 仍生成 DOCX，失败任务在报告中标注"代码执行失败：{错误信息}"，退出码 0
- 全部任务失败或文本生成失败 → 退出码 1

#### 4.2.4 input() 替换器增强

**问题**：原始方案中多个 Cell 共用 `_idx` 全局变量，前一个 Cell 消耗了 test_inputs 后，后续 Cell 的 `_idx` 未重置，导致读取到空值。此外，代码可能通过 `sys.stdin.readline()` 绕过替换器。

**解决方案**：

1. 每个 Cell 使用独立变量名（`_test_inputs_0`、`_test_inputs_1`...），避免跨 Cell 串扰
2. 同时替换 `sys.stdin.readline` 和 `sys.stdin.read`
3. System Prompt 中约束：代码中禁止重新导入 `input`

```python
# ====== 自动生成的 input 替换器（Cell 0 专用）======
_test_inputs_0 = ["50", "25", "abc", "37"]
_idx_0 = 0

def _mock_input_0(prompt=""):
    global _idx_0
    print(prompt, end="")
    if _idx_0 < len(_test_inputs_0):
        val = _test_inputs_0[_idx_0]
        _idx_0 += 1
        print(val)
        return val
    return ""

import sys
__builtins__.input = _mock_input_0
sys.stdin.readline = lambda: _mock_input_0("")
sys.stdin.read = lambda: "\n".join(_test_inputs_0)
# ====== 替换器结束，以下是原始代码 ======
```

```python
def extract_third_party_imports(code: str) -> list[str]:
```

1. Windows 字体文件路径：`C:\Windows\Fonts\NotoSansSC-VF.ttf`、`msyh.ttc`、`simsun.ttc`、`simhei.ttf`
2. 字体族名称：`Noto Sans CJK SC`、`Microsoft YaHei`、`SimSun`、`SimHei`
3. Linux fallback：`WenQuanYi Micro Hei`、`DejaVu Sans`
4. 系统默认字体

若全部失败，退回 Pillow 默认字体并写入 warning；默认字体缺少中文 glyph 时，图片中的中文会显示为乱码或方块，因此优先使用本机中文字体文件路径。

### 4.4 模板引擎

#### 4.4.1 书签定位（替代关键词匹配）

在 Word 模板中预先插入 4 个书签，脚本通过书签名称精确定位插入点，不依赖段落文本内容：

| 书签名称 | 注入内容 |
|----------|----------|
| `bookmark_objectives` | `report_data.objectives` |
| `bookmark_topics` | `report_data.topics`（逐条编号列表） |
| `bookmark_details` | `report_data.details` |
| `bookmark_code` | 代码任务标题 + 源代码文字段落 + 运行结果图片 + 题注 |

```python
from docx import Document

def fill_at_bookmark(doc: Document, bookmark_name: str, content):
    """在指定书签位置插入内容"""
    for paragraph in doc.paragraphs:
        for run in paragraph.runs:
            for bookmark in run._element.findall(
                './/{http://schemas.openxmlformats.org/wordprocessingml/2006/main}bookmarkStart'
            ):
                if bookmark.get(
                    '{http://schemas.openxmlformats.org/wordprocessingml/2006/main}name'
                ) == bookmark_name:
                    # 在书签所在段落之后插入内容
                    parent = paragraph._element.getparent()
                    index = list(parent).index(paragraph._element)
                    # 插入新段落
                    new_para = OxmlElement('w:p')
                    # ... 设置样式和内容
                    parent.insert(index + 1, new_para)
                    return
```

#### 4.4.2 图片题注格式

```

```

#### 4.4.2.1 代码与结果分离

第四部分“课程代码及执行结果”中，每个任务按以下顺序插入：

1. `任务 {i}：{任务名称}`
2. `任务描述：`，保留 LLM 阶段一识别出的原始任务描述，便于失败时人工补全
3. `源代码：`，随后以与正文一致的 Word 普通文本格式插入最终代码（包含自动修复后的代码）；若代码生成失败，则显示“未生成源代码”并保留失败原因，便于手动补报告
4. `运行结果：`，随后插入 PNG 图片；图片只包含 stdout/stderr 文本和可选 matplotlib 图形，不再包含源代码；插图宽度为 6.3 英寸以提高可读性
5. 图片题注 `图 {day}-{序号}  {任务名称}`

### 4.5 主控制器

#### 4.5.1 流水线步骤

```
try:
    Step 0: 加载 config.yaml，清空 temp/ 目录
    Step 1: 读取 daily_notes/{date_str}.txt，执行 rstrip() 清理
    Step 2: 阶段一 — LLM 提取结构化文本（TextData）
    Step 3: 阶段二 — 若 TextData.task_descriptions 非空，逐一调用 LLM 生成代码；JSON 失败时走纯代码抽取和本地保底模板
    Step 4: 代码沙箱 — 对每个 CodeTask:
        Step 4a: AST 扫描，检测危险模块导入
        Step 4b: 子进程隔离执行
        Step 4c: 若失败，自动修复循环（最多 3 次）
        Step 4d: 记录 ExecutionResult（成功/失败）
    Step 5: 图片渲染 — 逐任务渲染（含图形输出）
    Step 6: 模板填充 — 书签定位 + 部分失败也生成 DOCX
    Step 7: 写入日志文件 logs/{date_str}.log
finally:
    Step 8: 清理 temp/ 临时文件
```

## 6. 错误处理

### 6.1 错误码定义

| 错误类型 | 场景 | 处理策略 |
|----------|------|----------|
| `FileNotFoundError` | 指定日期的 TXT 不存在 | 写入日志，退出码 1 |
| `json.JSONDecodeError` | LLM 返回非标准 JSON | 阶段一重试后仍失败则退出；阶段二会继续尝试纯 Python 代码生成、代码抽取和本地保底模板 |
| `ConnectionError` | API 网络超时 | 重试 3 次，间隔 2s，仍失败则退出 |
| `RuntimeError` | 单个代码任务执行失败 | 标记 failed，继续执行其他任务 |
| `RuntimeError` | 全部代码任务失败 | 仍生成 DOCX，退出码 1 |
| `ValueError` | TextData 缺少必要字段 | 退出并打印缺失字段名 |
| `FontNotFoundError` | 无可用中文字体 | 退出并提示安装 `fonts-noto-cjk` |

### 6.2 日志规范

- 日志写入文件 `logs/YYYY-MM-DD.log`，方便定时任务运行后排查
- `INFO`：流水线步骤开始/完成、API 调用耗时、任务执行状态
- `WARNING`：`code_tasks` 为空、API 重试、单个任务失败
- `ERROR`：导致流程中断的异常

## 8. 测试用例

### 8.1 正常流程测试

**输入**：完整笔记 TXT，含交互式任务（猜数字游戏）

**预期**：
- 阶段一返回 `TextData`，objectives/topics/details 非空
- 阶段二返回 5 个 `CodeTask`，每个含 `test_inputs`
- 所有任务沙箱执行成功，input 替换器生效
- 输出 `.docx` 与参考样本结构一致

### 8.2 部分失败测试

**输入**：TXT 含 3 个代码任务，其中 1 个 LLM 生成代码有运行时错误

**预期**：
- 失败任务触发自动修复循环（最多 3 次），修复后仍失败则标记 failed
- 另 2 个任务正常执行
- 最终生成 DOCX，失败任务在报告中标注错误信息
- 退出码 0

### 8.3 危险代码拦截测试

**输入**：LLM 生成了 `import os; os.system("rm -rf /")` 的代码

**预期**：
- AST 扫描检测到 `os` 模块，标记任务为 failed
- 日志记录"检测到危险模块导入：os"
- 不执行，不阻断其他任务

**预期**：
- preamble 自动注入，拦截 `plt.show()`

**输入**：TXT 含猜数字游戏（需要 `input()` 交互，含异常处理）

**预期**：
- LLM 生成 `test_inputs` 覆盖正常数字、异常字符、汉字
- 每个 Cell 使用独立变量名，不串扰
- `sys.stdin.readline` 也被替换
- 输出与自然交互视觉效果一致
